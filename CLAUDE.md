## AI-DLC Operating Rules (read first)

This repository is being used for a governed AI-DLC exercise. These rules take
precedence over convenience and apply to every turn.

### Operating model
AI proposes, human governs. Claude accelerates analysis, test scaffolding, and
mechanical edits. The human owns scope, business meaning, and risk acceptance.

### Scope
- All work is confined to the `companyms` service. Other services are explicitly
  out of scope for this exercise.
- Goal: introduce the first real, behaviour-asserting tests, with focus on the
  Resilience4j fallback contract on the Feign client paths.

### Working discipline
- Make the smallest controlled change that achieves the goal. No drive-by refactors.
- Test-first: write the test, run it, confirm it asserts real behaviour, then stop.
- Tests must be pure unit tests using JUnit 5 + Mockito. Mock the repository and
  Feign clients. Do NOT depend on a running database, RabbitMQ, or network.
- Show diffs before applying any production-code change, and wait for explicit "apply".
- When a behaviour is a design decision (e.g. silent vs loud fallback, 404 vs 502),
  STOP and ask the human. Do not guess at business semantics.
- Flag any blast-radius increase (new dependency, new class, changed public
  signature) and justify it or drop it.

### Definition of Done (hard gates, in order — stop at first failure)
1. `cd companyms && mvn clean test` — green
2. New tests genuinely assert behaviour (not empty `contextLoads` stubs)
3. No production logic changed unless a test exposed a real bug (then record an ADR)
4. One ADR recorded in `docs/adr/` for the fallback-semantics decision

## Build & Run Commands

Each microservice is an independent Maven project. Run all commands from the service's subdirectory.

```bash
# Build (skip tests)
cd <service-dir>
mvn clean package -DskipTests

# Run tests
mvn test

# Run a single test class
mvn test -Dtest=CompanyServiceImplTest

# Run with Docker Compose (all services together)
docker-compose -f docker-compose.yaml up --build
```

### Required startup order (local dev)
1. `service_registry` (port 8761)
2. `config_server` (port 8080)
3. `companyms` / `jobms` / `reviewms` (ports 8081–8083) — in any order
4. `api_gateway` (port 8084)

RabbitMQ must be running for `companyms` and `reviewms` to function fully. Start one with Docker:
`docker run -d -p 5672:5672 -p 15672:15672 -e RABBITMQ_DEFAULT_USER=user -e RABBITMQ_DEFAULT_PASS=password rabbitmq:3-management`.

## Architecture

### Service Map

| Service | Port | Role |
|---|---|---|
| `service_registry` | 8761 | Eureka server — all services register here |
| `config_server` | 8080 | Spring Cloud Config Server; pulls config from external GitHub repo |
| `api_gateway` | 8084 | Spring Cloud Gateway; routes `/companies/**` → companyms, `/jobs/**` → jobms, `/reviews/**` → reviewms |
| `companyms` | 8081 | Company CRUD; aggregates jobs + reviews via Feign; owns company rating state |
| `jobms` | 8082 | Job CRUD; fetches parent company via Feign |
| `reviewms` | 8083 | Review CRUD; publishes review events to RabbitMQ |

### Inter-service Communication

**Synchronous (OpenFeign):**
- `companyms` → `jobms` via `JobClient` (get/delete jobs by companyId)
- `companyms` → `reviewms` via `ReviewClient` (get reviews by companyId)
- `jobms` → `companyms` via `CompanyClient` (get company by id)
- `reviewms` → `companyms` via `CompanyClient` (get company by id)

FeignClients use `url="${<service>.url:}"` so the URL can be overridden per profile; otherwise they rely on Eureka discovery (`name="JOBMS"` etc.).

**Asynchronous (RabbitMQ — DirectExchange):**
- `reviewms` publishes to `review.exchange` with routing keys `review.created`, `review.updated`, `review.deleted`
- `companyms` listens on `review.created.queue.companies`, `review.updated.queue.companies`, `review.deleted.queue.companies` and updates the company's aggregated rating
- `companyms` publishes to `company.exchange` with routing key `company.deleted` when a company is deleted
- `reviewms` listens on `company.deleted` queue and deletes all reviews for that company

All RabbitMQ messages use `Jackson2JsonMessageConverter` and `PERSISTENT` delivery mode.

### Within Each Business Microservice (companyms / jobms / reviewms)

```
bean/          – domain model / DTOs (passed over REST and Feign)
client/        – Feign client interfaces
controller/    – REST controllers + GlobalExceptionHandler
dao/           – Spring Data JPA repositories
entity/        – JPA entities (DB representation)
eventDTO/      – RabbitMQ event payload classes
exception/     – custom exceptions (e.g. CompanyNotFoundException)
mapper/        – MapStruct mappers (entity ↔ bean conversions)
messaging/     – RabbitMQ configuration, publishers, listeners
response/      – composite response objects (e.g. CompanyResponse with jobs + reviews)
service/       – service interfaces
service/impl/  – service implementations
```

### Resilience (Resilience4j)

Every Feign call is wrapped with `@CircuitBreaker`, `@Retry`, and `@RateLimiter` in the `*ClientServiceImpl` classes. Circuit breaker instances are named after the downstream service (`jobBreaker`, `reviewBreaker`). Fallback methods return empty lists for reads and throw `RuntimeException` for deletes. `CompanyNotFoundException`, `FeignException$NotFound`, and `IllegalArgumentException` are configured as ignored exceptions (they do not count toward the failure rate). Config lives in `application.properties` (slidingWindowSize=10, failureRateThreshold=50, max retry attempts=5, rate limiter 2/sec).

### Rating Aggregation (companyms)

`CompanyEntity` stores rating state. Updates from review events use a single atomic JPQL `UPDATE` in `CompanyRepository.updateRatingAtomically()` to avoid read-modify-write races under concurrent events. (The non-atomic read-modify-write version is preserved as comments in `CompanyServiceImpl`, showing the race was deliberately fixed.)

### Profiles & Databases

| Profile | Database | Notes |
|---|---|---|
| `dev` (default) | h2 on classpath, **not explicitly wired** in `application.properties` | Verify before assuming a working DB; tests should mock the repository |
| `docker` | PostgreSQL (`my_postgres_db:5432`) | `init/init.sql` creates the 3 databases on first run |
| `k8s` | PostgreSQL (Kubernetes StatefulSet) | Manifests in `k8s/` |

`spring.config.import=optional:configserver:...` — the `optional:` prefix means services start even if the config server is unreachable.

### Distributed Tracing

Zipkin runs on port 9411. All services set `management.tracing.sampling.probability=1.0` and use `micrometer-tracing-bridge-brave` + `zipkin-reporter-brave`. The `feign-micrometer` dependency propagates trace context across Feign calls.
